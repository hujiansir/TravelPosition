# 城市定位打卡功能 设计文档

| 项 | 值 |
|---|---|
| 日期 | 2026-06-14 |
| 状态 | 待评审 |

---

## 1. 背景与目标

进入小程序后展示**全国版图**，自动获取当前位置，将当前所在**地级市**以色块形式「点亮」，并作为足迹**持久化打卡**到当前登录用户。再次进入或换设备登录后，已点亮城市保持。

本功能在现有脚手架（仅有微信登录半成品 `WxAuthController` + 默认 index 页）之上构建，复用既有的 `Result`/`BusinessException`/`RedisUtil`/三层结构等约定。

---

## 2. 需求决策（已与用户确认）

| 维度 | 决策 |
|---|---|
| 点亮语义 | 打卡存档·持久化足迹（后端表 + User + 登录态） |
| 点亮粒度 | 地级市（约 330 个，存行政区划码 adcode） |
| 地图呈现 | 微信原生 `<map>` 组件（腾讯地图底图）+ DataV GeoJSON + polygons 填充 |
| 渲染策略 | 方案 A：完整版图底图（全国 330 市边界可见，未点亮灰、已点亮高亮） |
| 逆地理 | 前端 `qqmap-wx-jssdk` 的 `reverseGeocoder` |
| 鉴权方案 | Redis 会话 token（不透明 token + Redis 校验，非 JWT） |
| MVP 附加 | 足迹进度统计 + 城市点击交互 |
| MVP 不做 | 独立足迹列表页、分享足迹海报 |

---

## 3. 总体架构与端到端数据流

**鉴权模型**：登录生成 UUID token → `RedisUtil.set("login:token:{token}", userId, 7天)` → 请求头 `Authorization: {token}` → 拦截器校验。比 JWT 更契合微信会话语义（支持挤号/主动失效），且零新增依赖。

**端到端链路**：

```
启动  app.ts onLaunch → wx.login → POST /wx/login
       后端: openid 查/建 User → 生成 token 存 Redis → 返回 {openid, token}
       前端: token 存 storage + globalData

进入主页  pages/index onLoad（并行三件事）
  ┌─ ① 加载市级 GeoJSON
  │    本地文件缓存命中? ── 是 → 读取
  │                      ── 否 → DataV CDN 拉取 34 个省级 _full.json
  │                              → 合并为全国市级 FeatureCollection → 写本地文件缓存
  ├─ ② 拉取我的足迹  GET /footprint/list  (带 token)
  │    → 返回已点亮 adcode 集合 + litCount
  └─ ③ 定位打卡
       wx.getLocation(gcj02) → qqmap reverseGeocoder → {adcode, cityName, province}
       → POST /footprint/check-in {adcode, cityName, province, lng, lat}
          后端: upsert (user_id+adcode 唯一) → 返回 {isNew, firstCheckInTime, litCount}
       → 前端刷新该城市 polygon 为"已点亮" + 进度+1 + 轻提示

交互  点击地图 → 前端 point-in-polygon 定位城市
       → 命中且在足迹集合 → 显示点亮时间
       → 未在足迹 → 提示"还未到过此地"
```

**分层职责**：
- 前端：地图渲染、GeoJSON 加载/缓存、定位、逆地理、上报、交互展示。
- 后端：鉴权、足迹 CRUD、进度计数。**不做**逆地理、**不存**全国城市边界（那是地图资源，前端管）。
- 后端只认 `adcode`；城市名/经纬度由前端逆地理得到后随上报传入，后端只存不校验坐标。

## 4. 后端设计

沿用现有约定：三层结构（controller/service.impl/mapper/entity/dto）、`@TableName("travel.xxx")`、`Result<T>` 统一返回、`BusinessException(code,message)` 抛错（已支持自定义码，鉴权失败用 401）、MyBatis Plus `IService`/`BaseMapper`。

### 4.1 登录闭环补全（足迹功能前置）

现有 `WxAuthController.login` 为 TODO（仅返回 openid，未落库未签发 token），需补全：

| 文件 | 操作 | 内容 |
|---|---|---|
| `service/UserService.java` | 新增方法 | `User loginOrRegister(String openid, String unionid)` |
| `service/impl/UserServiceImpl.java` | 实现 | 调 `getByOpenid`，存在返回；不存在 `new User` 存 openid/unionid 后 `save` |
| `controller/WxAuthController.java` | 补全 login TODO | 调 `loginOrRegister` → 生成 token（UUID 去横线）→ **挤号**：若 `login:user:{userId}` 有旧 token 先删旧 → 写 `login:token:{token}=userId`(7天) + `login:user:{userId}=token`(7天) → 返回 `{openid, token}` |
| `common/UserContext.java` | **新增** | `ThreadLocal<Long>` 持有当前 userId；`getNotNull()` 为空抛 `BusinessException(401,"未登录")` |
| `common/AuthInterceptor.java` | **新增** | `preHandle`：取 `Authorization` 头 → 查 Redis → `UserContext.set(userId)`；查不到/过期抛 `BusinessException(401)`；`afterCompletion` 清理 ThreadLocal |
| `config/WebConfig.java` | 注册拦截器 | `addPathPatterns("/**").excludePathPatterns("/wx/**")` |

挤号（同一用户新登录踢掉旧 token）默认实现，属小程序常见安全需求。

### 4.2 足迹表 DDL（追加到 `src/main/resources/sql/init.sql`）

```sql
CREATE TABLE IF NOT EXISTS travel.city_footprint (
  id                    BIGSERIAL PRIMARY KEY,
  user_id               BIGINT      NOT NULL,
  adcode                VARCHAR(6)  NOT NULL,   -- 城市行政区划码
  city_name             VARCHAR(64) NOT NULL,
  province_name         VARCHAR(64),
  lng                   NUMERIC(10,6),          -- 打卡坐标，仅存档不参与判定
  lat                   NUMERIC(10,6),
  first_check_in_time   TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_check_in_time    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  create_time           TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  update_time           TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  deleted               SMALLINT NOT NULL DEFAULT 0,
  CONSTRAINT uk_user_adcode UNIQUE (user_id, adcode)   -- 幂等兜底
);
CREATE INDEX IF NOT EXISTS idx_footprint_user ON travel.city_footprint(user_id);
COMMENT ON TABLE travel.city_footprint IS '用户城市打卡足迹';
-- 复用已有 travel.update_modified_column() 触发器函数，挂同名触发器 update_footprint_modtime
DROP TRIGGER IF EXISTS update_footprint_modtime ON travel.city_footprint;
CREATE TRIGGER update_footprint_modtime
    BEFORE UPDATE ON travel.city_footprint
    FOR EACH ROW
    EXECUTE FUNCTION travel.update_modified_column();
```

唯一约束保证并发打卡幂等。业务上足迹不删除（逻辑删除字段保留仅为与项目约定一致）。

### 4.3 实体 / DTO / VO

- `entity/CityFootprint`：字段对应上表，`@TableLogic deleted`、`@TableName("travel.city_footprint")`、`@TableId(type=IdType.AUTO)`。坐标用 `BigDecimal`。
- `dto/CheckInRequest`：`@NotBlank adcode`、`@NotBlank cityName`、`provinceName`、`lng`、`lat`（JSR-303 校验注解）。
- `vo/FootprintVO`：`{adcode, cityName, provinceName, firstCheckInTime, lastCheckInTime}`（列表/详情复用）。
- `vo/FootprintListVO`：`{litCount, adcodes:List<String>, items:List<FootprintVO>}`。
- `vo/CheckInVO`：`{isNew, firstCheckInTime, litCount}`。

**进度分母不在后端存**：`total` 由前端 GeoJSON 的 `features.length` 决定（最准），后端只返回 `litCount`，避免魔法常量与城市字典表。

### 4.4 Service / Controller

`FootprintService`：
- `FootprintListVO listMine()` — `UserContext.getNotNull()` 取 userId，查该用户全部足迹。
- `CheckInVO checkIn(CheckInRequest req)` — **upsert**：查 `(userId, adcode)`，存在则 `lastCheckInTime=now`、`isNew=false`；不存在则 insert、`isNew=true`。并发依赖唯一约束 + 捕获 `DuplicateKeyException` 后转查询兜底。

> detail 接口不做：点击交互用的点亮时间已在 `list` 的 `items` 中，前端本地即可展示，无需额外请求（YAGNI）。

`FootprintController`（`@RequestMapping("/footprint")`，全部需登录）：

| 方法 | 路径 | 请求 | 返回 data |
|---|---|---|---|
| GET | `/footprint/list` | — | `FootprintListVO` |
| POST | `/footprint/check-in` | `CheckInRequest` | `CheckInVO` |

### 4.5 边界与约定

- 后端只认 adcode，不校验坐标真实性（职责在前端地图）。
- 鉴权失败统一 `code=401`。
- 全部沿用 `Result`/`BusinessException`/`GlobalExceptionHandler`，无新异常体系。

### 5.2 主页流程（pages/index）

```
onLoad:
  await auth.ensureLogin()                       // 确保 token 就绪
  geojson = await geo.loadCityGeoJSON()          // 本地文件优先，否则 CDN 拉取合并
  list  = await footprintApi.list()              // 我的足迹 adcode 集合
  this.renderMap(geojson, list.adcodes)          // 生成 polygons → setData
  this.tryCheckIn()                              // 定位+逆地理+上报（异步，失败不阻塞渲染）

onShow:
  // MVP 不自动重新打卡，避免频繁上报；提供「重新定位」按钮手动触发

renderMap(geojson, litAdcodes):
  polygons = geojson.features.map(f => toPolygon(f, litAdcodes.has(f.adcode)))
  // 已点亮: fillColor 高亮色(#FFC107,α.6)  未点亮: 描边灰、fill 透明
```

### 5.3 GeoJSON 加载与缓存（`utils/geo/geojson.ts`）

- **缓存介质**：合并后约 5–10MB，用**本地文件**（`wx.env.USER_DATA_PATH/city_geojson.json`）——`wx.setStorageSync` 上限 10MB 装不下且会挤占业务缓存。
- **加载顺序**：本地文件存在且**校验通过**（可解析为合法 GeoJSON FeatureCollection 且 features 数 > 0）→ 直接读；否则并行拉取 34 个省级 `_full.json` → 合并为一个 FeatureCollection → 写本地文件 → 返回。
- **更新**：MVP 首次拉取后长期使用本地文件；预留版本号字段，后续可加"启动时检查版本、按需重拉"。
- **抽稀**：DataV `_full.json` 精度中等，MVP 先用原始数据；`simplify.ts` 预留，实测卡顿再启用。

### 5.4 点亮渲染（map 组件 + polygons）

- 每个 GeoJSON feature 的 `geometry.coordinates`（`[lng,lat]` 数组）→ 转成 map 的 `polygon.points`（`{latitude,longitude}`）。
- 已点亮/未点亮用不同 `fillColor` 区分，`litAdcodes` 用 `Set<string>`，O(1) 查询。
- **性能风险**：330 个 polygon 一次性 `setData` 数据量较大，中低端机可能首次渲染卡顿。缓解：① 按需精简点数；② 必要时分批 `setData`。MVP 先全量，预留优化点。

### 5.5 逆地理与上报（`tryCheckIn`）

```
wx.getLocation(type:'gcj02') → {lat,lng}
  失败/拒绝 → 引导 wx.openSetting；主页占位"开启定位以点亮城市"
qqmap.reverseGeocoder({location:{latitude:lat,longitude:lng}})
  → ad_info.adcode(区县6位) + ad_info.city(城市名)
cityAdcode = 区县adcode 取前4位+"00"   （地级市码；特例见第 6 节兜底）
footprintApi.checkIn({adcode:cityAdcode, cityName, province, lng, lat})
  → {isNew, litCount}   isNew=true 时该城市 polygon 切高亮 + 进度+1 + 轻提示
```

### 5.6 点击交互

`map bindtap` 拿点击坐标 → `pip.pointInPolygons(lng,lat, geojson)` 定位城市 → 命中且在足迹集合 → 直接用本地 list 数据（已含 `firstCheckInTime`）展示点亮时间，无需额外请求；未在足迹 → 提示"还未到过此地"。map 无原生 polygon 点击事件，故用前端 ray-casting 判断（数据已在手，无需二次网络）。

### 5.7 鉴权集成（`request.ts` 改 + `auth.ts`）

- `request.ts`：每次请求从 storage 读 token 写入 `Authorization` 头。
- 响应 `code===401`：调 `auth.handle401()`——清旧 token → `wx.login` 重登拿新 token → 重试原请求一次；仍失败则 toast"登录已过期，请重进"。

### 5.8 进度展示

主页顶部：`已点亮 {litCount} / {total}`，`total` 来自 `geojson.features.length`（最准）。可加细进度条。

## 6. 数据资源与外部依赖

| 资源 | 说明 | 落点 |
|---|---|---|
| **qqmap key** | 在 [lbs.qq.com](https://lbs.qq.com) 申请「小程序」类 key，供 `reverseGeocoder` 用 | `config/index.ts` |
| **DataV GeoJSON** | base：`https://geo.datav.aliyun.com/areas_v3/bound/`；先取 `100000_full.json` 拿 34 省 adcode，再拉每个省 `{省码}_full.json`（含该省市级 features） | `data/province-adcodes.ts` + `utils/geo/geojson.ts` |
| **adcode 映射兜底** | 腾讯返回**区县级** adcode；地级市码 = 前 4 位 + "00"。直辖市成立；省直辖县级市（潜江/天门/济源等）有特例 → **双保险**：同时用 `ad_info.city` 城市名在本地 GeoJSON 按 `name` 匹配兜底 | `utils/geo/geojson.ts` |

**坐标系**：微信 `map` 组件与 `getLocation(gcj02)` 均为 GCJ-02；腾讯 `reverseGeocoder` 返回 GCJ-02，与 map 一致，无需转换。DataV 边界坐标系需实现时实测确认（见第 11 节开放问题）。

## 7. 权限与错误处理

**隐私合规（发布阻塞项）**：
- `app.json` 声明 `"requiredPrivateInfos": ["getLocation"]`（小程序新规，否则 `wx.getLocation` 不可用）。
- 配置 `permission.scope.userLocation` 的 `desc` 说明用途。

**错误场景矩阵**：

| 场景 | 处理 |
|---|---|
| 定位未授权/拒绝 | 引导 `wx.openSetting`；主页占位"开启定位以点亮城市"；已点亮城市仍正常渲染 |
| `getLocation` 失败 | toast「定位失败」，不阻塞地图 |
| `reverseGeocoder` 失败 | toast「无法识别当前位置」，本次不打卡 |
| `check-in` 网络失败 | 静默重试 1 次，仍失败提示「打卡失败，点击重试」 |
| GeoJSON 拉取失败且无本地缓存 | 退路：纯底图 + toast「地图数据加载失败」；有缓存则用缓存 |
| `401` 未登录/过期 | `auth.handle401`：重登 + 重试原请求一次；仍失败 toast |
| 登录未完成就进主页 | `auth.ensureLogin()` 在 onLoad 阻塞等待 `app.ts` 的 wxLogin 完成 |

**上报节制**：onLoad 打卡一次 + 手动「重新定位」按钮，不做后台/定时定位（耗电、接口压力）。

## 10. 非目标（YAGNI）

明确不在 MVP 范围：
- 独立足迹列表页（后续可加 `pages/footprint`）。
- 分享足迹海报（生成图片分享，成本高，后续迭代）。
- 好友/排行等社交功能。
- 后台定时定位、行程轨迹记录。
- 后端城市字典表与 adcode 合法性强校验（前端 GeoJSON 已隐含合法集）。
- GeoJSON 边界抽稀（预留钩子，性能不足再启用）。

## 11. 待实现验证的开放问题

1. **DataV GeoJSON 坐标系**：实现时实测确认是 WGS84 还是 GCJ-02。若为 WGS84，城市级填充因边界跨度大、偏移视觉影响小可接受；需严格对齐时启用 `coord-transform.ts`。
2. **330 polygon 首次渲染性能**：中低端机实测，若卡顿则按"精简点数 / 分批 setData / 启用抽稀"递进优化。
3. **省直辖县级市 adcode 特例**：潜江/天门/神农架/济源/新疆兵团等，验证"前 4 位 + 00"规则是否与 DataV 市级 feature 对齐，必要时用城市名匹配兜底。
4. **MyBatis Plus 逻辑删除与唯一约束的潜在冲突**：本场景足迹不删除，冲突不触发；若未来支持删除足迹再重打，需将 `deleted` 纳入唯一约束或改用软删 + 业务层查重。
